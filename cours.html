Étape 1 — Réorganisation de l'arborescence — 2026-05-02

Organiser par feature, pas par type technique

« Pourquoi devais-je créer un package notes et un package editor ? »

Quand on commence un projet, il y a deux façons d'organiser ses fichiers :

Organisation par type technique (à éviter pour les apps modernes)

ui/
├── screens/       ← tous les écrans en vrac
├── viewmodels/    ← tous les ViewModels en vrac
└── components/    ← tous les composants en vrac

Organisation par feature (ce qu'on a fait)

ui/
├── notes/         ← TOUT ce qui touche à la liste de notes
│   ├── NotesScreen.kt
│   └── (à venir) NotesViewModel.kt, NoteCard.kt, ...
├── editor/        ← TOUT ce qui touche à l'éditeur de note
│   ├── EditorScreen.kt
│   └── (à venir) EditorViewModel.kt, TranscriptionState.kt, ...
└── theme/         ← le thème visuel (couleurs, typo)

Pourquoi c'est mieux ?

  1. Cohésion : quand tu travailles sur la feature « notes », tu ouvres un seul dossier et tout est là. Tu n'as pas à sauter entre screens/, viewmodels/, components/. Les fichiers qui changent ensemble vivent ensemble.
  2. Découplage visible : si ui/editor/ importe quelque chose de ui/notes/, c'est un signe louche → ces deux features devraient être indépendantes. Le fait de les voir dans des dossiers séparés rend cette règle évidente.
  3. Scalabilité : à 30 écrans, un dossier screens/ plat devient illisible. Avec une organisation par feature, chaque dossier reste petit et focalisé.
  4. C'est ce que Google recommande dans ses docs officielles d'architecture Android (Guide to app architecture).

Pourquoi sortir les écrans de ui/theme/screens/

Le squelette de départ avait ui/theme/screens/ — c'est-à-dire que les écrans étaient dans le sous-package du thème. Or :

  • Le thème (ui/theme/) = les couleurs, la typographie, le mode sombre. Du visuel pur, partagé par toute l'app.
  • Les écrans = des features fonctionnelles (afficher des notes, éditer une note).

Mettre les écrans dans theme/, c'est dire « les écrans dépendent du thème ». Faux : c'est l'inverse, les écrans utilisent le thème (via MaterialTheme.colorScheme), mais ils n'en sont pas une sous-partie. C'était une erreur d'organisation du squelette à corriger.

Le fichier NotesScreen.kt

« À quoi servent les fichiers que j'ai mis dedans ? »

Rôle : afficher la liste des notes de l'utilisateur. C'est l'écran d'accueil de l'app (premier onglet de la bottom nav future).

État actuel : juste un Scaffold qui affiche « Bienvenue sur SmartNotes ». C'est un placeholder — il sera enrichi à l'étape 9 (LazyColumn de cards avec titre/date/aperçu).

Anatomie du code actuel

@Composable
fun NotesScreen(navController: NavHostController) {
    Scaffold {
        Text(text = "Bienvenue sur SmartNotes")
    }
}
ÉlémentExplication

ScaffoldComposant Material 3 qui donne la structure standard d'un écran : barre du haut, contenu, FAB, barre du bas. Pour l'instant on n'utilise pas ces slots, on met juste du texte au centre.

Le fichier EditorScreen.kt

Rôle : afficher l'éditeur de note (création + édition). C'est l'écran qui s'ouvre quand tu cliques sur « + » ou sur une note existante.

État actuel : pareil, placeholder avec « Ajouter un article ». Sera enrichi à l'étape 10 (champ titre + champ contenu + bouton enregistrer).

Même anatomie que NotesScreen — @Composable + Scaffold + texte. C'est juste un autre écran indépendant, dans son propre package ui/editor/ pour bien le séparer de la feature « notes ».

Le rôle du navController

C'est la pièce qui relie les deux écrans. Imagine navController comme une télécommande que les écrans se passent. Quand NotesScreen veut t'envoyer dans l'éditeur, il fait :

navController.navigate(Routes.ADD)

→ Android cherche dans le NavGraph quel composable correspond à Routes.ADD, et l'affiche. Tu reverras tout ça à l'étape 3 quand on branchera vraiment la navigation (avec en plus la bottom nav et les 3 onglets).

Récap de l'étape 1 :

  • On a sorti les écrans de ui/theme/screens/ (mauvais endroit, ils ne dépendent pas du thème).
  • On a créé un package par feature : ui/notes/ et ui/editor/.
  • Chaque package contient l'écran principal de la feature, et plus tard ses ViewModels et composants spécifiques.
  • Les écrans communiquent via le navController et les routes définies dans navigation/Routes.kt.

Étape 2 — Application + AppContainer — 2026-05-02

Qu'est-ce qu'un singleton ?

« C'est quoi un singleton ? »

Définition : un singleton est un objet dont il n'existe qu'une seule instance pendant toute la durée de vie du programme. Tout le monde qui le demande reçoit la même instance.

Pourquoi c'est utile ?

Imagine ta DB Room. Ouvrir une connexion à la base, c'est coûteux (allocation mémoire, chargement du schéma, etc.). Si chaque écran de ton app créait sa propre instance de SmartNotesDatabase, tu aurais 5 connexions ouvertes en parallèle, plus une fuite mémoire à chaque navigation. Une seule DB partagée par toute l'app = une seule connexion, un seul cache, un seul état cohérent.

Comment on en fait un en Kotlin ?

FormeUsage typique
object MonObjet { ... } Singleton « pur » du langage. Initialisé au premier accès. Pas de paramètres, pas de Context.
Une classe instanciée une seule fois dans Application.onCreate() Quand tu as besoin d'un Context ou de paramètres au constructeur. C'est notre cas avec AppContainer.
companion object { val instance by lazy { ... } } Quand tu veux un accès « global » via MaClasse.instance. Plus rare.

Dans notre projet, AppContainer n'est pas un object Kotlin parce qu'il a besoin d'un Context à la construction. On en garantit l'unicité par discipline : on ne le crée qu'à un seul endroit (dans SmartNotesApp.onCreate()), et tout le monde le lit depuis là.

Parallèle JS : c'est comme exporter export const db = createDb() dans un module — comme JS ne charge le module qu'une fois, tu obtiens un singleton de fait.

Pourquoi un dossier di/ ?

« Pourquoi appeler le dossier di ? »

di = abréviation standard pour Dependency Injection (injection de dépendances).

C'est quoi DI ?

C'est un pattern où, au lieu qu'une classe crée elle-même les objets dont elle a besoin, on les lui donne de l'extérieur.

Exemple sans DI :

class NotesViewModel {
    private val repo = NoteRepository(SmartNotesDatabase.create(...))
    // le ViewModel décide tout seul comment construire son repo
}

Avec DI :

class NotesViewModel(private val repo: NoteRepository) {
    // le ViewModel reçoit son repo, il ne sait pas comment il a été construit
}

Pourquoi c'est mieux ?

  • Testabilité : en test, tu peux injecter un repo bidon (fake) sans toucher la DB.
  • Découplage : le ViewModel ne dépend plus de la DB ni de Room directement, juste de l'interface du repo.
  • Lifecycle : un seul repo partagé entre tous les ViewModels, au lieu de N copies.

L'AppContainer est l'endroit où on construit toutes ces dépendances une seule fois, et où on les expose. C'est notre « boîte à outils centrale ». Donc le mettre dans un dossier di/ est une convention claire pour les futurs lecteurs (toi-même dans 6 mois) : « ah, tout ce qui touche à l'assemblage des dépendances est ici ».

Si tu trouves le nom obscur, tu pourrais l'appeler container/ ou core/. Mais di/ est la convention Android la plus répandue — autant prendre l'habitude.

Qu'est-ce qu'une init / un onCreate ?

« C'est quoi une init ? »

« Initialiser » un objet = lui donner sa valeur de départ. Quand tu écris var x: Int = 5, tu initialises x à 5. Sans cette init, x n'a pas de valeur et tu ne peux pas l'utiliser.

Pour les classes, il y a deux moments d'init possibles :

  1. À la construction : quand tu écris val container = AppContainer(context), le constructeur tourne et l'objet est prêt.
  2. Plus tard, dans une méthode onCreate() : c'est ce qu'on fait pour SmartNotesApp.container. Pourquoi pas à la construction ? Parce qu'au moment où Android construit SmartNotesApp, le applicationContext n'est pas encore disponible — Android nous le passe juste avant d'appeler onCreate(). Donc on ne peut pas initialiser container plus tôt.

D'où le lateinit var : c'est la promesse Kotlin « je te jure que je l'initialiserai avant qu'on s'en serve, laisse-moi juste le déclarer maintenant sans valeur ». Si tu oublies l'init et que quelqu'un essaie de lire container, l'app crashe avec un message clair lateinit property container has not been initialized.

À quoi sert AppContainer ?

« À quoi sert le fichier AppContainer ? »

L'image générale : ton app est un restaurant. La DB Room = le frigo. Le client Gemini = la ligne téléphonique vers l'IA. Tu veux un seul exemplaire de chaque pour tout le restaurant. AppContainer est la boîte à outils centrale qui contient ces objets partagés.

Pour l'instant, la boîte est vide. C'est ça qui peut perturber : « à quoi sert un fichier vide ? »

Réponse : on pose la boîte maintenant pour pouvoir y mettre des outils plus tard sans tout casser. Aux étapes suivantes, on va y ajouter :

  • étape 6 : val database = SmartNotesDatabase.create(context) ← le frigo
  • étape 7 : val noteRepository = NoteRepository(database.noteDao()) ← l'employé qui sait quoi faire avec le frigo
  • étape 23 : val geminiClient = GeminiClient(apiKey) ← la ligne téléphonique vers l'IA

À chaque ajout, n'importe quel écran de l'app pourra faire :

val app = LocalContext.current.applicationContext as SmartNotesApp
val notes = app.container.noteRepository.getAllNotes()

Traduction : « Hé bâtiment, donne-moi ta boîte à outils, et passe-moi l'employé du frigo. »

Pourquoi le faire maintenant si c'est vide ?

Parce que c'est plus simple d'ajouter une étagère dans une boîte qui existe déjà que de tout réorganiser plus tard. Si on attendait l'étape 6 pour créer SmartNotesApp + AppContainer en même temps que Room, tu aurais 4 nouveaux concepts à digérer d'un coup. Là on isole la plomberie maintenant, on la teste (l'app se lance ? oui), et aux étapes suivantes on aura juste à remplir la boîte.

package io.github.marievial2.smartnotes.di

    // Pour l'instant vide.
    // Aux étapes 6+ : database, noteRepository, etc.

À quoi sert SmartNotesApp ?

« À quoi sert le fichier SmartNotesApp ? »

SmartNotesApp est le bâtiment du restaurant. Android le construit avant d'ouvrir la première Activity, et il existe tant que ton app tourne.

C'est la classe que tu déclares dans le manifest avec android:name=".SmartNotesApp". Sans cette ligne dans le manifest, Android ne sait même pas que ta classe existe et utilise un bâtiment générique vide à la place.

Concrètement, dans son onCreate() (= « moment où le bâtiment ouvre ses portes »), elle fait une seule chose : créer le AppContainer.

Le code de la classe (étape 2)

package io.github.marievial2.smartnotes

import android.app.Application
import io.github.marievial2.smartnotes.di.AppContainer

class SmartNotesApp : Application() {

    lateinit var container: AppContainer
        private set

    override fun onCreate() {
        super.onCreate()
        container = AppContainer(applicationContext)
    }
}
LigneExplication
class SmartNotesApp : Application()Hérite de Application. Le : Application() = « hérite et appelle le constructeur sans argument ». Équivalent du extends Java.
lateinit var container: AppContainerlateinit = « je promets de l'initialiser avant qu'on l'utilise, laisse-moi la déclarer sans valeur ». var obligatoire avec lateinit.
private setTout le monde peut lire container, mais seul SmartNotesApp peut l'écrire. Empêche un code extérieur de casser le singleton.
override fun onCreate()Redéfinit la méthode héritée. Android l'appelle automatiquement, une seule fois, juste avant la première Activity.
super.onCreate()Appel à la classe-mère. Obligatoire. Sans ça, Android crashe au lancement.
container = AppContainer(applicationContext)On crée le container en lui passant applicationContext. applicationContext est dispo gratuitement parce que SmartNotesApp est un Context.

Modif du AndroidManifest.xml

Il faut ajouter android:name=".SmartNotesApp" sur la balise <application>. Le . au début est un raccourci « préfixe avec le namespace du module ».

<application
    android:name=".SmartNotesApp"
    android:allowBackup="true"
    ...>

Qu'est-ce qu'une Activity ?

« C'est quoi une activity ? »

Une Activity = un écran de ton app. Point.

Quand tu utilises une app sur ton téléphone, ce que tu vois à un instant T, c'est une Activity. Quand tu touches « Voir le détail » et qu'un nouvel écran s'ouvre, c'est une autre Activity (ou alors la même Activity qui change de contenu — c'est notre cas avec Compose).

Exemples concrets

  • WhatsApp : la liste des conversations = 1 Activity. Tu cliques sur une conv → la conversation ouverte = 1 autre Activity.
  • Une app météo : l'écran principal = 1 Activity. L'écran « Réglages » = 1 autre Activity.
  • SmartNotes : pour l'instant on n'a qu'une seule Activity, MainActivity. C'est elle qui contient toute l'app. À l'intérieur, c'est Compose qui change ce qui est affiché (les « écrans » Compose ≠ Activities Android).

Pourquoi c'est important ?

Une Activity a une vie courte et imprévisible :

  • Tu tournes ton téléphone en mode paysage → Android détruit l'Activity et la recrée (vrai !).
  • L'utilisateur appuie sur retour → l'Activity est détruite.
  • Une autre app prend le focus (un appel arrive) → l'Activity est mise en pause, peut-être détruite.

Donc tu ne peux rien stocker de permanent dans une Activity. Ce qui est dedans peut disparaître à tout moment.

Analogie web : une Activity, c'est comme une page HTML dans le navigateur. Tu changes de page → l'ancienne est détruite, ses variables JS partent en fumée. Pareil ici.

Qu'est-ce qu'un Context ?

« C'est quoi un contexte ? »

Un Context est un objet qui te donne accès à Android lui-même (le système d'exploitation).

Quand tu codes en JS pour le web, tu as window, document, navigator qui te donnent accès au navigateur. Sur Android, tout passe par un Context.

Exemples de ce qu'on fait avec un Context

  • Lire un texte traduit : context.getString(R.string.app_name) ← demande au système la version dans la langue de l'utilisateur.
  • Ouvrir un fichier privé à l'app : context.openFileInput("notes.json") ← le système te donne le chemin où ton app a le droit d'écrire.
  • Ouvrir la base Room : Room.databaseBuilder(context, ...) ← Room a besoin de savoir où poser le fichier .db.
  • Lancer un autre écran : context.startActivity(intent).
  • Obtenir un service système : context.getSystemService(Context.CONNECTIVITY_SERVICE) ← pour vérifier la connexion internet.

Sans Context, tu ne peux rien faire d'Android-spécifique.

Image : vois ça comme ton badge d'employé. Sans badge, tu ne peux rien faire dans le bâtiment. Avec, tu accèdes aux portes que ton rôle autorise.

Les deux types de Context

« C'est quoi un contexte d'activity ? »

Il existe plusieurs sortes de Context, mais en pratique, deux qui comptent :

applicationContext — le badge « global »

  • Qui le fournit : la classe Application (donc notre SmartNotesApp).
  • Durée de vie : tant que ton app tourne. Ne meurt jamais avant la fermeture de l'app.
  • Quand l'utiliser : pour tout ce qui doit survivre à la rotation d'écran ou aux changements d'écran. Singletons, base de données, clients réseau.

Le Context d'une Activity — le badge « écran courant »

  • Qui le fournit : une Activity (chaque Activity est un Context par héritage).
  • Durée de vie : courte. Détruit à la rotation, au retour arrière, etc.
  • Quand l'utiliser : pour tout ce qui est lié à l'écran courant : afficher une AlertDialog, un Toast, démarrer une autre Activity.

Pourquoi cette distinction est CRITIQUE

Code-piège (à ne PAS faire) :

class AppContainer(context: Context) {
    val database = Room.databaseBuilder(context, ...).build()
}

// Dans ton Activity :
val container = AppContainer(this)  // ← "this" = l'Activity, donc Context d'Activity ❌

L'utilisateur tourne son téléphone → Android veut détruire l'Activity pour la recréer. Mais il ne peut pas : ton AppContainer (singleton qui vit pour toujours) tient encore une référence vers cette Activity. La mémoire de l'Activity ne peut pas être libérée. C'est ce qu'on appelle une fuite mémoire (memory leak). Tournes 50 fois, tu as 50 Activities mortes en mémoire.

Solution : passer applicationContext à la place. Lui, il vit pour toujours de toute façon, donc personne à libérer.

val container = AppContainer(applicationContext)  // ✅ pas de fuite

La chaîne complète au démarrage de l'app

  1. L'utilisateur lance l'app.
  2. Android instancie SmartNotesApp (parce qu'on l'a déclaré dans le manifest).
  3. Android appelle SmartNotesApp.onCreate().
  4. À l'intérieur, on appelle AppContainer(applicationContext). On lui passe nous-mêmes (puisque SmartNotesApp est un Context).
  5. AppContainer stocke ce Context dans son champ private val context.
  6. Plus tard (étape 6), Room en aura besoin pour créer son fichier de DB.
  7. Android instancie MainActivity et lance le premier écran.
  8. Tu touches l'écran, tu tournes le téléphone, peu importe : MainActivity peut être détruite et recréée 100 fois, le AppContainer et son Context ne bougent pas.

Récap en 3 phrases :

  • Activity = un écran de l'app, à durée de vie courte.
  • Context = badge d'accès à Android, indispensable pour tout faire d'Android-spécifique.
  • applicationContext = le badge global qui vit pour toujours, à utiliser pour les singletons. Le Context d'Activity = le badge de l'écran courant, à utiliser uniquement pour des trucs liés à l'écran.

Étape 3 — Bottom navigation : routes, placeholders, écrans Compose — 2026-05-03

L'objet Routes : pourquoi centraliser les routes

« Pourquoi un fichier Routes.kt avec un object au lieu de mettre les chaînes directement dans le code ? »

Une route, c'est l'identifiant stable d'une destination de l'app. Quand tu écris navController.navigate("notes"), la chaîne "notes" est la route de l'écran Notes.

Le piège des chaînes en dur

Si tu écris "notes" à 12 endroits différents dans le code, il suffit qu'à un endroit tu tapes "note" (sans s) :

  • le compilateur Kotlin ne dit rien (c'est juste une chaîne valide),
  • l'app compile,
  • elle plante à l'exécution en disant « no destination found for route 'note' ».

Avec une constante Routes.NOTES = "notes", si tu tapes Routes.NOTE par erreur, le compilateur t'engueule tout de suite : « unresolved reference NOTE ». Tu corriges avant même de lancer l'app.

object Routes : un singleton léger

object en Kotlin = un singleton du langage. Pas de constructeur, pas de Routes(), on accède directement aux membres : Routes.NOTES. C'est l'équivalent d'une classe avec uniquement des membres static en Java. Idéal pour un sac de constantes thématiquement liées.

object Routes {
    const val NOTES = "notes"
    const val CATEGORIES = "categories"
    const val TAGS = "tags"
    const val EDITOR_NEW = "editor/new"
    const val EDITOR_EDIT = "editor/{noteId}"

    fun editorEdit(noteId: String): String = "editor/$noteId"
}
ÉlémentRôle
const valConstante évaluée à la compilation. Plus efficace que val simple, mais limité aux types primitifs et String.
NOTES, CATEGORIES, TAGSLes 3 destinations de la bottom nav.
EDITOR_NEWOuvrir l'éditeur en mode création (pas d'ID).
EDITOR_EDITGabarit pour ouvrir l'éditeur en mode édition (avec un ID dynamique).

Les placeholders {xxx} dans une route

Regarde la constante EDITOR_EDIT = "editor/{noteId}". Le {noteId} est un paramètre de route — un trou dans l'URL que Compose Navigation va remplir avec une valeur réelle au moment de la navigation.

Pourquoi un placeholder ?

L'éditeur peut afficher n'importe quelle note. Tu n'as pas envie de déclarer une route par note (ce serait absurde — il y en aura des centaines). Une seule route avec un trou : "editor/{noteId}". Côté Compose Navigation, à l'arrivée sur l'écran, on lit la valeur réelle :

val noteId = backStackEntry.arguments?.getString("noteId")

La règle du nom à 3 endroits

Le nom du placeholder (noteId) doit correspondre exactement entre :

  1. la constante de route ("editor/{noteId}"),
  2. la déclaration côté NavGraph (à venir en 3.4),
  3. le code qui lit la valeur (à venir en étape 11).

Si tu écris {noteId} dans la route mais arguments?.getString("note_id") côté lecture, tu obtiendras null en silence. Pas d'erreur de compilation. Bug à l'exécution.

Analogie web : c'est comme un paramètre dynamique dans une URL Express ou React Router : /users/:userId. Même principe.

Le helper editorEdit(noteId) : gabarit vs URL réelle

Distinction essentielle :

ChaîneRôle
"editor/{noteId}"Gabarit (template). On le déclare au NavGraph pour dire « il existe une destination de cette forme ».
"editor/abc-123"URL réelle. C'est ce qu'on passe à navController.navigate(...) quand on veut vraiment naviguer.

La fonction helper fait juste la substitution :

fun editorEdit(noteId: String): String = "editor/$noteId"

Usage côté code (à venir étape 11) :

navController.navigate(Routes.editorEdit("abc-123"))
// → navigue vers "editor/abc-123"

Pourquoi String et pas UUID en paramètre ? Une URL est toujours du texte. La conversion UUID ↔ String se fait aux frontières (juste avant navigate et juste après getString("noteId")). Cohérent avec la correction faite à l'étape 1.

L'annotation @Composable

« Pourquoi cette annotation au-dessus de chaque écran ? »

L'annotation @Composable dit au compilateur Kotlin : « cette fonction décrit de l'UI, pas un calcul classique ».

@Composable
fun CategoriesScreen(navController: NavHostController) { ... }

Conséquences

  • Une fonction @Composable ne peut être appelée que depuis une autre fonction @Composable (ou depuis setContent { ... } de l'Activity).
  • Elle peut être réexécutée plusieurs fois par le système Compose. C'est ce qu'on appelle la recomposition : quand une donnée affichée change, Compose ré-exécute la fonction pour produire la nouvelle UI.
  • Elle ne renvoie rien (Unit). Sa « sortie », c'est ce qu'elle déclare via les composables qu'elle appelle (Text, Box, etc.).

Si tu oublies l'annotation, l'IDE crache : « Functions which invoke @Composable functions must be marked with the @Composable annotation ».

Mental model : pense une fonction @Composable comme une template d'UI déclarative, pas comme un calcul impératif. Tu décris « ce qui doit être à l'écran », pas « les étapes pour le dessiner ». Compose s'occupe du dessin et des mises à jour.

Le Modifier : équivalent CSS de Compose

« C'est quoi un Modifier ? »

Un Modifier est une chaîne d'instructions qu'on accroche à un composable pour lui dire comment se comporter : sa taille, ses marges, son fond, s'il est cliquable, s'il scroll, etc.

Box(
    modifier = Modifier
        .fillMaxSize()         // 1. occupe toute la place
        .padding(16.dp)        // 2. laisse 16dp de marge intérieure
        .background(Color.Red) // 3. fond rouge
)

Chaque appel (fillMaxSize(), padding(...), etc.) est un maillon de la chaîne. Le composable applique tout dans l'ordre.

L'analogie web/CSS

ComposeCSS équivalent
Modifier.fillMaxSize()width: 100%; height: 100%
Modifier.padding(16.dp)padding: 16px
Modifier.background(Color.Red)background-color: red
Modifier.clickable { ... }onclick="..." + cursor: pointer
Modifier.size(100.dp)width: 100px; height: 100px

Modifier ≈ l'équivalent CSS+événements DOM, mais cousu directement dans le code Kotlin avec autocomplétion, type-checking, et impossibilité de te tromper de nom de propriété.

Pourquoi pas des paramètres directs sur chaque composable ?

Imagine si chaque composable avait 40 paramètres pour la taille, le padding, le fond, etc. Chaque nouveau composable maison devrait re-déclarer tout ça. Pas de réutilisation.

Avec Modifier, tu as un seul mécanisme universel partagé par tous les composables. Tu apprends 10-15 modifiers de base et tu peux les appliquer à n'importe quoi : Box, Text, Column, Image, ton propre composable maison… tout.

L'ordre compte (subtil mais important)

Les modifiers sont appliqués de gauche à droite. Tu peux obtenir des résultats différents selon l'ordre :

// Padding PUIS background
//   → marge à l'extérieur du fond (carré rouge plus petit avec de l'air autour)
Modifier.padding(16.dp).background(Color.Red)

// Background PUIS padding
//   → fond plus grand, contenu décalé à l'intérieur
Modifier.background(Color.Red).padding(16.dp)

Règle mentale : chaque modifier transforme la zone vue par le suivant.

Les modifiers les plus courants

ModifierEffet
fillMaxSize(), fillMaxWidth(), fillMaxHeight()« occupe toute la place dispo » sur 1 ou 2 axes
size(W, H) ou size(100.dp)taille fixe
padding(...)marge intérieure
background(color)fond coloré
clickable { ... }rend cliquable + ripple visuel
align(Alignment.X)dans Box/Column/Row, alignement individuel d'un enfant
weight(1f)dans Column/Row, distribution proportionnelle de l'espace

Pas besoin de tous les retenir : Android Studio les propose en autocomplétion dès que tu tapes Modifier..

Convention « modifier en 1er paramètre optionnel »

Tu remarqueras que tous les composables Compose ont un paramètre modifier: Modifier = Modifier. C'est une convention :

@Composable
fun MonComposable(
    // 1. Paramètres obligatoires
    text: String,
    onClick: () -> Unit,

    // 2. Le modifier en premier paramètre optionnel,
    //    avec valeur par défaut Modifier (= chaîne vide)
    modifier: Modifier = Modifier,

    // 3. Le reste des paramètres optionnels
    color: Color = Color.Black,
)

Quand tu écriras tes propres composables, fais pareil. C'est la convention officielle Google.

Alignment vs contentAlignment & conventions de casse Kotlin

« C'est quoi contentAlignment et Alignment ? Et pourquoi Alignment a une majuscule ? »

Deux choses distinctes dans la même ligne

Box(
    modifier = Modifier.fillMaxSize(),
    contentAlignment = Alignment.Center
    //  ↑ paramètre        ↑ valeur d'un type
)
ÉlémentNatureCasse
contentAlignmentparamètre du composable Box (« où aligner le contenu »)lowerCamelCase
Alignmenttype (un peu comme un enum)PascalCase
Alignment.Centervaleur au sein de ce type (« centré »)Conformément au type

La convention de nommage Kotlin

C'est la même règle qu'en Java, et c'est universel dans presque tous les langages typés :

CatégorieConventionExemples
Classes, interfaces, objets, typesPascalCaseAlignment, NotesScreen, NavHostController, Modifier
Variables, paramètres, fonctions, propriétéslowerCamelCasecontentAlignment, navController, fillMaxSize, noteId
Constantes const val top-level ou dans un objectUPPER_SNAKE_CASERoutes.NOTES, EDITOR_EDIT
Packagestout.en.minusculesio.github.marievial2.smartnotes.ui.notes

C'est purement visuel — le compilateur s'en fiche. Mais en lisant Alignment.Center, tu sais immédiatement :

  • Alignment (majuscule) → c'est un type, on accède à quelque chose qu'il expose.
  • Center (majuscule aussi) → c'est une valeur (constante) au sein de ce type.
  • contentAlignment (minuscule) → c'est un paramètre, donc une donnée qu'on passe.

Les valeurs disponibles dans Alignment

Alignment.TopStart      Alignment.TopCenter      Alignment.TopEnd
Alignment.CenterStart   Alignment.Center         Alignment.CenterEnd
Alignment.BottomStart   Alignment.BottomCenter   Alignment.BottomEnd

9 alignements 2D (3 verticaux × 3 horizontaux), exposés comme des constantes typées. On les utilise comme un enum.

Pourquoi Start/End plutôt que Left/Right ? Pour gérer les langues qui s'écrivent de droite à gauche (arabe, hébreu). En LTR (français, anglais), Start = Left. En RTL, Start = Right. Compose le gère automatiquement. Tu n'as jamais à utiliser Left/Right (ça existe mais c'est un piège).

contentAlignment — pourquoi ce nom précis

Le paramètre s'appelle contentAlignment parce que Box peut contenir plusieurs enfants empilés. contentAlignment dit « voici l'alignement par défaut du contenu » — c'est-à-dire des enfants du Box, pas du Box lui-même.

Box(
    modifier = Modifier.fillMaxSize(),
    contentAlignment = Alignment.Center  // valeur par défaut pour tous les enfants
) {
    Text("Au centre")
    Text(
        text = "En haut à droite",
        modifier = Modifier.align(Alignment.TopEnd)  // override individuel
    )
}

Donc : contentAlignment = alignement global par défaut pour tout le contenu du Box. Modifier.align(...) = override pour un enfant spécifique.

Mini-récap :

  • contentAlignment = paramètre du Box (lowerCamelCase parce que c'est un paramètre).
  • Alignment = type (PascalCase parce que c'est un type), qui expose 9 valeurs prédéfinies (Center, TopStart, etc.).
  • Règle de casse Kotlin : PascalCase pour les types, lowerCamelCase pour le reste, UPPER_SNAKE_CASE pour les constantes const val.
  • Start/End au lieu de Left/Right pour le support RTL automatique.

Les 3 niveaux de la navigation : Routes / NavGraph / navController

« Grâce à composable(), j'ai fait le lien entre le nom des routes et leur écran ? »

Oui, exactement. Pour avoir une navigation qui marche, il faut 3 fichiers/objets distincts qui jouent chacun un rôle clair. Bien les séparer mentalement évite beaucoup de confusion.

  3. Le NavHost voit le changement, va chercher dans sa table (les composable("categories") { ... }) le bloc correspondant.

remember et rememberNavController() : survivre aux recompositions

« Pourquoi rememberNavController() et pas juste NavController(...) ? »

Une fonction @Composable peut être ré-exécutée des dizaines de fois par seconde (recomposition) — par exemple chaque fois qu'un état affiché change, Compose ré-exécute la fonction pour produire la nouvelle UI.

Le piège

Si tu écrivais :

@Composable
fun Mauvais() {
    val navController = NavController(...)  // ❌
    // ...
}

Un nouveau navController serait créé à chaque recomposition → la pile de navigation serait perdue à chaque clic. L'app ne saurait jamais où elle est.

La solution : remember { ... }

remember est le mécanisme Compose pour « crée cette valeur la première fois, et garde-la d'une recomposition à l'autre ».

@Composable
fun Bon() {
    val navController = remember { NavController(...) }  // ✅
    // ...
}

rememberNavController() est juste un raccourci pré-câblé pour le navController :

val navController = rememberNavController()
// équivalent grosso modo à :
val navController = remember { NavHostController(...) }

La règle générale à retenir

Toute valeur qui doit survivre aux recompositions doit être enveloppée dans un remember { ... }. Sans, elle est recalculée à chaque fois.

Tu vas voir remember partout en Compose : remember { mutableStateOf(...) }, rememberCoroutineScope(), rememberSaveable, etc. Familiarise-toi avec.

Et la rotation d'écran ?

Attention : remember survit aux recompositions, mais pas à la destruction-recréation de l'Activity (ex: rotation de l'écran). Pour ça, il faut rememberSaveable ou un ViewModel. À voir plus tard quand on aura besoin de persister un état utilisateur.

currentBackStackEntryAsState() : observer la route courante

Comment savoir, à tout instant, quelle route est affichée — pour pouvoir mettre en évidence le bon onglet de la bottom nav ?

Le navController a deux APIs

APIRéactif ?Quand l'utiliser
navController.currentBackStackEntry❌ nonLecture ponctuelle (callback, méthode non-Composable).
navController.currentBackStackEntryAsState()✅ oui — retourne un StateDans une fonction @Composable — Compose recompose quand la route change.

Usage typique

val backStackEntry by navController.currentBackStackEntryAsState()
val currentRoute = backStackEntry?.destination?.route

Le by à la place du =

currentBackStackEntryAsState() retourne un State<NavBackStackEntry?>. Pour lire la valeur, tu pourrais faire state.value. Mais Kotlin propose un delegate avec by qui te donne directement la valeur sans .value :

// Sans delegate
val backStackEntryState = navController.currentBackStackEntryAsState()
val currentRoute = backStackEntryState.value?.destination?.route

// Avec delegate "by" (équivalent, plus lisible)
val backStackEntry by navController.currentBackStackEntryAsState()
val currentRoute = backStackEntry?.destination?.route

Pour que by fonctionne sur un State, il faut l'import androidx.compose.runtime.getValue. Android Studio l'ajoute automatiquement.

Pourquoi un ? partout

Quand l'app démarre, il y a un instant très bref où aucune destination n'est encore active → backStackEntry peut être null. Le ? dit à Kotlin « si c'est null, propage le null ; sinon continue ». Sans le ?, Kotlin refuse de compiler par sécurité.

Le Scaffold parent et innerPadding

Scaffold : la structure standard d'un écran

Scaffold est le composable Material 3 qui donne la structure standard d'un écran avec des "slots" :

Règle : un seul Scaffold parent

Dans une app à navigation centralisée (bottom nav, drawer, etc.), il y a un seul Scaffold parent dans MainActivity. Les écrans individuels mettent leur contenu directement, sans réinventer la chrome (top bar, bottom bar). Sinon tu te retrouves avec des Scaffolds imbriqués, deux barres potentielles, des paddings qui s'empilent.

(Plus tard, un écran spécifique peut quand même avoir son propre Scaffold s'il a besoin d'une TopAppBar particulière — par exemple NotesScreen avec son menu de tri. C'est l'exception.)

Le piège de l'innerPadding

Scaffold(
    modifier = Modifier.fillMaxSize(),
    bottomBar = { /* la bottom nav */ }
) { innerPadding ->
    Box(modifier = Modifier.padding(innerPadding)) {
        AppNavGraph(navController)
    }
}

Le Scaffold calcule l'espace restant pour le contenu une fois que les barres sont posées. Il te passe ce calcul via innerPadding (le (innerPadding) -> ... du lambda).

Tu dois absolument utiliser ce innerPadding sur le contenu (via Modifier.padding(innerPadding)), sinon le contenu va passer sous la bottom bar et tu ne verras pas les derniers éléments. C'est le piège le plus classique avec Scaffold.

Le piège du stub TODO créé par Android Studio

Bug rencontré pendant l'étape 3 : l'app crashe au lancement avec NotImplementedError: An operation is not implemented: Not yet implemented.

Le scénario

Tu écris dans MainActivity.kt :

AppNavGraph(navController)

Mais tu n'as pas encore ajouté l'import qui pointe vers la vraie fonction (celle du fichier navigation/NavGraph.kt). Android Studio voit une référence non résolue et te propose plusieurs quick-fixes en surlignage rouge, dont :

  • ✅ « Import » — ajoute l'import io.github.marievial2.smartnotes.navigation.AppNavGraph en haut du fichier. C'est ce qu'on veut.
  • ❌ « Create function 'AppNavGraph' » — crée une fonction stub vide dans le fichier courant.

Si tu cliques par erreur sur le mauvais quick-fix (ou si tu valides avec Entrée trop vite), Android Studio te génère ça en bas du fichier :

@Composable
fun AppNavGraph(x0: NavHostController) {
    TODO("Not yet implemented")
}

Pourquoi ça crashe l'app

Quand une fonction est définie dans le même fichier que là où on l'appelle, elle a priorité sur tout import. Donc AppNavGraph(navController) appelle ce stub local, pas le vrai. Et la fonction TODO(...) de la stdlib Kotlin lève une NotImplementedError à chaque appel — elle n'existe que pour marquer du code à compléter.

Comment repérer le bug

  • Symptôme : l'app compile (BUILD SUCCESSFUL) mais crashe immédiatement à l'ouverture.
  • Logcat : FATAL EXCEPTION avec NotImplementedError: An operation is not implemented: Not yet implemented.
  • Indice : le paramètre s'appelle x0 (Android Studio nomme automatiquement les paramètres générés x0, x1, etc. — un nom humain ne ressemblerait jamais à ça). Si tu vois x0 quelque part, c'est presque toujours le signe d'un quick-fix mal placé.

Comment corriger

  1. Supprimer entièrement la fonction stub.
  2. Ajouter le bon import en haut du fichier.

Comment l'éviter à l'avenir

Renommage SmartNotesAppRoot → MainScreen

« Pourquoi le composable racine ne peut pas s'appeler SmartNotesApp tout court ? »

Une app Compose forme un arbre de composables : MainActivity appelle un composable racine, qui appelle Scaffold, qui appelle AppBottomBar et AppNavGraph, qui appellent CategoriesScreen, etc. Il faut un nom pour ce composable au sommet de l'arbre.

  • @Composable fun SmartNotesApp() — composable UI.

Légal en Kotlin (namespaces différents), mais source de confusion énorme à la lecture et imports ambigus.

            }

La pile de navigation — analogie de la pile d'assiettes

« Je pense que je ne comprends pas parfaitement cette histoire de pile. »

Une pile (en anglais stack) en informatique, c'est exactement comme une pile d'assiettes :

  • Tu empiles une nouvelle assiette → elle se met sur le dessus.
  • Tu dépiles → tu prends celle du dessus, et la suivante devient visible.
  • Tu ne peux toucher qu'à celle du dessus.

En Android, chaque écran que tu visites est une assiette. La pile garde l'historique de ta navigation.

Exemple concret avec SmartNotes

Lancement de l'app :

Pile : [Categories]
        ↑ écran de départ, seul sur la pile
        Compose affiche : Categories (le sommet)

L'utilisateur clique l'onglet « Notes » :

Pile : [Categories, Notes]
                    ↑ Notes empilé par-dessus
                    Compose affiche : Notes

Il clique une note pour l'éditer :

Pile : [Categories, Notes, Editor]
                            ↑ Editor empilé
                            Compose affiche : Editor

Il appuie sur le bouton retour du téléphone :

Pile : [Categories, Notes]
                    ↑ Editor dépilé
                    Compose affiche : Notes (à nouveau)

Le bouton retour Android, physiquement, c'est juste « dépiler ». C'est pour ça qu'il « te ramène en arrière » : tu reviens à l'assiette précédente.

C'est le NavController (créé par rememberNavController()) qui maintient cette pile. Ses méthodes : navigate("notes") empile, popBackStack() dépile.

NavController = télécommande, NavHost = téléviseur

« Je crois que je ne comprends pas à quoi sert NavHost (rappelle moi où il est). »

Où est NavHost ?

Il est dans navigation/NavGraph.kt :

@Composable
fun AppNavGraph(navController: NavHostController) {
    NavHost(                           // ← le voici
        navController = navController,
        startDestination = Routes.CATEGORIES
    ) {
        composable(Routes.CATEGORIES) { CategoriesScreen(navController) }
        composable(Routes.TAGS)       { TagsScreen(navController) }
        composable(Routes.NOTES)      { NotesScreen(navController) }
        composable(Routes.EDITOR_NEW) { EditorScreen(navController) }
        composable(Routes.EDITOR_EDIT){ EditorScreen(navController) }
    }
}

Et il est appelé dans MainActivity.kt dans le slot content du Scaffold :

Box(modifier = Modifier.padding(innerPadding)) {
    AppNavGraph(navController)        // ← l'appel
}

Ce que fait NavHost

C'est le composable qui regarde le sommet de la pile, et qui affiche l'écran correspondant.

  • Tu lui donnes un navController (pour qu'il puisse lire la pile).
  • Tu lui donnes un startDestination (la première assiette à empiler quand l'app démarre).
  • Tu lui donnes une table de correspondance « route ↔ écran » via les composable("...") { ... }.

À chaque changement de pile :

  1. NavHost regarde le sommet (ex : "notes").
  2. Il cherche dans sa table le bloc associé (ex : composable(Routes.NOTES) { NotesScreen(...) }).
  3. Il affiche ce bloc.

Métaphore clé : NavHost = le téléviseur. NavController = la télécommande. Le NavController dit « passe à la chaîne notes » en empilant "notes", et le NavHost se met à afficher NotesScreen.

Qu'est-ce qu'un NavBackStackEntry ? Pourquoi nullable ?

« C'est quoi backStackEntry ? »

Un NavBackStackEntry, c'est l'objet qui représente une assiette dans la pile. Pour un écran donné dans la pile, il contient :

  • destination : l'objet qui décrit cette destination (sa route, son label éventuel, ses arguments attendus). C'est de là qu'on tire .route.
  • arguments : les paramètres passés à cet écran (ex : pour editor/{noteId}, le noteId réel comme "42").
  • D'autres champs qu'on n'utilise pas encore (savedStateHandle, lifecycle…).

Donc backStackEntry.destination.route te donne la string "notes", "categories", etc. — la route de l'écran actuellement au sommet de la pile.

currentBackStackEntryAsState() veut dire littéralement : « l'entrée actuellement en haut de la pile, exposée comme un State observable ».

Pourquoi NavBackStackEntry? (avec un ?) ?

Au tout premier rendu de l'app, le NavHost n'a pas encore eu le temps de pousser sa première destination sur la pile. Il y a un micro-instant où la pile est vide → currentBackStackEntry est null.

Chronologie au lancement :

  1. Compose démarre MainScreen.
  2. navController est créé. Sa pile est vide à ce moment-là.
  3. On demande currentBackStackEntryAsState() → la pile est vide → renvoie null.
  4. Plus tard (quelques µs après) : NavHost s'exécute, voit startDestination = CATEGORIES, empile "categories".
  5. La pile change → currentBackStackEntryAsState se met à jour → backStackEntry n'est plus null → recomposition.

D'où le ? sur NavBackStackEntry? : il y a ce micro-instant entre 2 et 4 où c'est null. Le code doit gérer ce cas :

val currentRoute = backStackEntry?.destination?.route
//                              ↑               ↑
//                              safe-call : si null, le résultat est null,
//                              sans plantage

State<T> : le journal abonnable de Compose

« Je ne comprends pas State<T>. »

Le problème que ça résout

En Compose, ta fonction @Composable est rejouée chaque fois que l'UI doit se rafraîchir. Mais Compose ne va pas tout rejouer en boucle 60 fois par seconde — ce serait du gâchis. Il a besoin de savoir précisément quand rejouer, et quoi rejouer. State<T> est la réponse.

L'analogie : un abonnement à un journal

Imagine que State<Int> soit un abonnement à un journal qui contient un seul nombre.

  • Si tu lis ce journal dans une fonction @Composable, Compose note : « ah, ce composable est abonné à ce journal ».
  • Quand le contenu du journal change (quelqu'un publie un nouveau numéro), Compose envoie un coup de fil à tous les composables abonnés : « hé, votre journal a changé, refaites votre rendu ».
  • Les composables qui n'ont jamais lu ce journal ne sont pas dérangés.

C'est ça, la recomposition ciblée.

Concrètement dans MainScreen

val backStackEntry by navController.currentBackStackEntryAsState()
//                    └──────────────────────────────────────────┘
//                    Cette fonction renvoie un State<NavBackStackEntry?>
//                    = un journal qui publie l'entrée actuelle au sommet de la pile.
  • Tant que tu es sur Categories, le journal contient entrée(route="categories").
  • Tu cliques Notes → la pile change → le journal publie un nouveau numéro entrée(route="notes").
  • Compose voit que MainScreen lit ce journal (via backStackEntry) → il rejoue MainScreen.
  • Du coup currentRoute recalcule à "notes", showBottomBar recalcule, le Scaffold se redessine — tout ça tout seul.

Différence avec un val normal

val x = 5                      // figé, ne change jamais
val x = mutableStateOf(5)      // un journal, abonnable, déclenche des recompositions

Si à la place de currentBackStackEntryAsState() on avait écrit currentBackStackEntry (la version non-State), on aurait eu la valeur à l'instant T, mais sans abonnement → pas de recompo automatique quand la pile change → la bottom bar resterait figée. C'est pour ça qu'on utilise toujours la variante ...AsState() quand on veut réagir au changement.

Mnémo : tout ce qui finit en AsState() ou qui est créé par mutableStateOf est un journal abonnable. Compose se charge du reste.

Le delegate by : sucre syntaxique sur State.value

Sans by, tu écrirais :

val backStackEntryState = navController.currentBackStackEntryAsState()  // State<NavBackStackEntry?>
val backStackEntry = backStackEntryState.value                          // NavBackStackEntry?

Avec by :

val backStackEntry by navController.currentBackStackEntryAsState()      // NavBackStackEntry?

C'est exactement la même chose, mais by te fait sauter le .value à chaque accès. C'est ce qu'on appelle un property delegate Kotlin.

Le rôle de l'import getValue

L'import androidx.compose.runtime.getValue en haut du fichier est précisément ce qui permet à by de fonctionner sur un State<T>. Sans cet import, le compilateur refuse le by. (Pour la version mutable avec =, on importe aussi setValue.)

Règle simple :

  • = ...AsState() → tu manipules .value à la main.
  • by ...AsState() → tu accèdes directement à la valeur, comme une variable normale.

Dans 95% des cas en Compose, on utilise by — c'est plus lisible.

Whitelist vs blacklist : pourquoi showBottomBar liste les routes autorisées

val showBottomBar = currentRoute in listOf(
    Routes.NOTES,
    Routes.CATEGORIES,
    Routes.TAGS
)

L'opérateur in

in est un opérateur Kotlin qui s'écrit comme en français : « est-ce que currentRoute est dans cette liste ? ». Il renvoie un Boolean. Équivalent de listOf(...).contains(currentRoute).

Donc showBottomBar est true si on est sur Notes, Catégories ou Tags. Sinon false (Editor, ou écran inconnu, ou null au tout premier instant).

Pourquoi une whitelist plutôt qu'une blacklist ?

On aurait pu écrire l'inverse :

// Blacklist (à éviter)
val showBottomBar = currentRoute != Routes.EDITOR_NEW
                 && currentRoute != Routes.EDITOR_EDIT
ApprocheComportement par défaut quand on ajoute un écran
Blacklist« Afficher PARTOUT, sauf sur Editor. » Demain, tu ajoutes Settings, CategoryDetail, TagFilter → la bottom bar y apparaîtra par erreur. Tu dois penser à les rajouter dans la blacklist.
Whitelist (notre choix)« Afficher SEULEMENT sur ces 3 onglets racines. » Tu ajoutes un nouvel écran → la bottom bar n'y apparaît pas par défaut. Tu dois explicitement l'ajouter à la whitelist si tu le veux.

La whitelist est plus sûre par défaut. Elle exprime clairement : « la bottom bar appartient aux trois onglets racines, point ». C'est l'esprit Material : la bottom nav est une nav de premier niveau, pas un truc qui traîne sur tous les écrans.

Principe général : préférer une whitelist à une blacklist quand l'ajout d'un nouveau cas doit, par sécurité, exclure plutôt qu'inclure.

Slot Compose = lambda passé en paramètre

Scaffold(
    modifier = Modifier.fillMaxSize(),
    bottomBar = {
        if (showBottomBar) {
            AppBottomBar(navController, currentRoute)
        }
    }
) { innerPadding -> ... }

Subtilité importante : bottomBar n'est pas un composable, c'est un paramètre auquel tu passes un lambda de type @Composable () -> Unit.

Pourquoi un lambda et pas une référence directe ?

Parce que Compose veut le contrôle sur quand et où exécuter ton code de rendu. Tu lui donnes une « recette » (le lambda), et il l'exécute lui-même au bon moment, dans le bon contexte de mise en page (le slot du bas, avec les bonnes contraintes de taille). Il peut décider de la sauter, de la rejouer, etc.

Le if à l'intérieur du lambda

Si showBottomBar est false, le if ne fait rien → le lambda ne contient aucun composable → le slot bottomBar est effectivement vide → Scaffold n'occupe pas d'espace en bas → la bottom bar disparaît, et le contenu s'étend pour récupérer la place.

Recompo automatique du slot

Quand tu navigues vers Editor :

  1. La pile change, donc backStackEntry change (rappel : c'est un State observable, cf. #state-journal).
  2. MainScreen est recomposé.
  3. currentRoute recalcule à "editor/new".
  4. showBottomBar recalcule à false.
  5. Scaffold rejoue son slot bottomBar → cette fois le if est faux → vide.
  6. La bottom bar disparaît visuellement.

Tout ça sans une ligne de logique impérative (« masque la bottom bar »). Tu décris ce qui doit être affiché en fonction de l'état, et Compose fait la mise à jour. C'est le paradigme déclaratif.

Le trailing lambda en Kotlin

En Kotlin, quand le dernier paramètre d'une fonction est un lambda, tu peux le sortir des parenthèses. C'est ce qu'on appelle un trailing lambda (« lambda final »).

La signature de Scaffold (simplifiée)

fun Scaffold(
    modifier: Modifier,
    topBar: @Composable () -> Unit,
    bottomBar: @Composable () -> Unit,
    ...,
    content: @Composable (PaddingValues) -> Unit  // ← dernier paramètre
)

Les deux écritures équivalentes

Sans trailing lambda (verbeux) :

Scaffold(
    modifier = Modifier.fillMaxSize(),
    bottomBar = { ... },
    content = { innerPadding -> Box(...) { AppNavGraph(navController) } }
)

Avec trailing lambda (idiomatique) :

Scaffold(
    modifier = Modifier.fillMaxSize(),
    bottomBar = { ... }
) { innerPadding ->
    Box(...) { AppNavGraph(navController) }
}

Le bloc { innerPadding -> ... } qui suit la parenthèse fermante est le slot content, sorti des parenthèses pour la lisibilité.

Tu reconnais un trailing lambda à une accolade ouvrante { qui suit immédiatement la parenthèse fermante ) d'un appel de fonction. Très courant en Compose et dans les builders Kotlin.

innerPadding et le Box wrapper

) { innerPadding ->
    Box(modifier = Modifier.padding(innerPadding)) {
        AppNavGraph(navController)
    }
}

innerPadding: PaddingValues — qu'est-ce que c'est ?

Quand le Scaffold dessine sa bottom bar (mettons 80dp de haut), il sait que ton contenu ne doit pas passer dessous (sinon ton texte serait caché). Il calcule donc un PaddingValues qui contient ces marges (top, bottom, start, end) et te le donne en paramètre du slot content, à toi de l'appliquer à ta racine de contenu.

Si tu oublies d'appliquer ce padding, ton contenu passera derrière la bottom bar (ou la status bar) — bug visuel classique.

Pourquoi un Box autour de AppNavGraph ?

  • Un Box est le composable de mise en page le plus simple : il empile ses enfants en superposition (z-order).
  • Ici on n'a qu'un enfant (AppNavGraph), donc le Box sert juste de support pour appliquer le Modifier.
  • Tu pourrais te passer du Box et appliquer le modifier directement sur AppNavGraph si on lui ajoutait un paramètre Modifier, mais comme on ne lui en a pas mis, le Box est un wrapper commode.

C'est une pratique courante : « j'ai besoin d'appliquer un Modifier mais le composable enfant n'expose pas de paramètre Modifier → je l'enveloppe dans un Box. »

Scaffold parent vs scaffold par écran

Question : pourquoi un seul Scaffold dans MainScreen qui englobe le NavHost, plutôt qu'un Scaffold dans chaque écran ?

Option A — Scaffold par écran

Chaque écran (NotesScreen, CategoriesScreen, TagsScreen…) aurait son propre Scaffold avec sa propre bottom bar. La bottom bar serait alors à l'intérieur du NavHost.

Option B — Scaffold parent (notre choix)

Un seul Scaffold dans MainScreen englobe le NavHost. La bottom bar est en dehors du NavHost, frère du contenu, pas enfant.

Côté animation de transition

NavHost applique une animation par défaut (fondu/slide) sur ce qu'il contient.

  • Option A : la bottom bar est dans le NavHost → l'animation s'applique aussi à elle. Elle glisse avec l'écran qui sort, et la bottom bar du nouvel écran arrive avec lui. Visuellement : la bottom bar « tremble » ou fait un saut à chaque changement d'onglet. L'œil perçoit ça comme un glitch — il s'attend à un repère stable.
  • Option B : la bottom bar est en dehors du NavHost → l'animation s'applique uniquement au contenu. La bottom bar reste parfaitement immobile, l'écran glisse sous elle. C'est l'effet « collée en bas » qu'on voit dans Gmail, YouTube, Instagram.

Côté Compose / recomposition

Recomposition ≠ destruction. Quand currentRoute change :

  • Option B : AppBottomBar est le même nœud dans l'arbre Compose. Il est recomposé avec le nouveau currentRoute, mais l'instance reste. Compose retrouve le nœud à la même position et lui passe les nouveaux paramètres. Coût minimal.
  • Option A : la bottom bar est dans l'écran qui sort → cet écran est démonté (sortie de la pile/animation). La bottom bar de cet écran est détruite. Celle du nouvel écran est montée (création de nouveaux nœuds Compose). Coût plus élevé, et surtout : impossible de garder un état dans la bottom bar entre deux écrans (puisqu'elle est recréée).

Côté code

  • Option A : code de la bottom bar dupliqué dans chaque écran (DRY violé), ou bien factorisé dans un composable réutilisable mais appelé partout.
  • Option B : un seul appel à AppBottomBar, dans MainScreen. La logique de visibilité (whitelist showBottomBar) est centralisée au même endroit.

Pattern Material recommandé : Scaffold parent. La bottom bar est un repère stable de l'app, pas un élément d'écran. Elle vit donc au-dessus du NavHost dans l'arbre Compose, et survit aux navigations.

2026-05-08 — Syntaxe d'un composable — pause avant le cours sur les lambdas

Anatomie d'un appel composable : parenthèses et accolades

Quand tu vois Scaffold(...) { ... }, Box { ... }, Text("Bonjour"), tu vois toujours la même chose : un appel de fonction Kotlin. Décortiquons la syntaxe une fois pour toutes.

Un composable, c'est une fonction Kotlin annotée

@Composable
fun MonComposable(param1: Type1, param2: Type2) {
    // corps
}

L'annotation @Composable dit au compilateur Compose : « cette fonction participe au système de recomposition ». Mais à part ça, c'est une fonction Kotlin normale. Tu l'appelles comme une fonction normale :

MonComposable(valeur1, valeur2)

Entre les parenthèses : les arguments

Comme pour n'importe quelle fonction Kotlin. On y met les valeurs que la fonction attend, dans l'ordre déclaré, ou avec leur nom (paramètres nommés). En Compose, on retrouve typiquement quatre familles d'arguments :

  • Données à afficher : Text(text = "Bonjour"), Icon(imageVector = Icons.Filled.Folder).
  • Modifier : Modifier.fillMaxSize(), Modifier.padding(8.dp) — taille, placement, décorations (cf. article #modifier).
  • Callbacks (lambdas) : onClick = { ... }, onValueChange = { newText -> ... } — « quand l'utilisateur fait X, exécute ça ».
  • Slot lambdas : bottomBar = { ... }, icon = { ... }, label = { ... } — « voici ce que tu dois afficher dans cette zone » (cf. article #slot-lambda).

Une lambda ({ ... }) est un argument comme un autre. Tu pourrais tout mettre entre parenthèses :

Scaffold(modifier = Modifier.fillMaxSize(), bottomBar = { AppBottomBar(...) }, content = { padding -> ... })

C'est valide. Juste illisible.

Entre les accolades qui suivent : le sucre syntaxique « trailing lambda »

Kotlin a une règle générale (pas spécifique à Compose) : si le dernier paramètre d'une fonction est une lambda, tu peux la sortir des parenthèses.

// Avec tout dans les parenthèses :
Scaffold(modifier = ..., bottomBar = { ... }, content = { padding -> ... })

// Avec la trailing lambda sortie :
Scaffold(modifier = ..., bottomBar = { ... }) { padding ->
    // ce qui était dans content = { ... }
}

Et si la lambda est le seul paramètre, tu peux même vider les parenthèses :

Box {
    Text("dans le box")
}
// strictement équivalent à :
Box(content = { Text("dans le box") })

Pourquoi c'est massivement utilisé en Compose

Parce que la plupart des composables containers (Box, Column, Row, Card, Scaffold, Surface, Button…) ont comme dernier paramètre content: @Composable () -> Unit — « la fonction qui décrit ce qui est à l'intérieur ».

Du coup les arbres Compose deviennent visuels :

Column {
    Text("titre")
    Row {
        Icon(...)
        Text("sous-texte")
    }
    Button(onClick = { ... }) {
        Text("Valider")
    }
}

Visuellement, les accolades dessinent l'arbre. Mais sous le capot c'est toujours : « passe une lambda comme dernier argument ».

Récap appliqué à notre Scaffold de MainScreen

Scaffold(
    modifier = Modifier.fillMaxSize(),     // arg classique
    bottomBar = {                          // arg lambda (slot, dans les parens)
        if (showBottomBar) AppBottomBar(...)
    }
) { innerPadding ->                       // dernier arg lambda, sorti en trailing
    Box(modifier = Modifier.padding(innerPadding)) {
        AppNavGraph(navController)
    }
}
  • Les parenthèses contiennent deux arguments : modifier et bottomBar.
  • Le troisième argument (content) est sorti en trailing lambda — c'est le { innerPadding -> ... }.
  • Le innerPadding -> indique que cette lambda reçoit un paramètre (le PaddingValues que Scaffold lui passe).

En une phrase : parenthèses = arguments classiques ; accolades qui suivent = le dernier argument de type lambda, sorti pour la lisibilité. Et c'est tout.

Lambdas Kotlin — 2026-05-09

Qu'est-ce qu'une lambda ? Graphie et rôle

« J'aimerais comprendre ce que c'est, pourquoi cette graphie et ce que ça fait. »

Définition : une lambda, c'est une fonction sans nom. C'est exactement ça, pas plus mystérieux. Pour le voir, voici la même idée écrite deux fois :

// Forme 1 : fonction nommée
fun doubler(x: Int): Int {
    return x * 2
}

// Forme 2 : lambda (= la même chose, sans nom)
{ x: Int -> x * 2 }

Les deux représentent le même comportement (« prends un entier, renvoie son double »). La seule différence, c'est que la première a un nom (doubler) et qu'on peut l'appeler plus tard ailleurs ; la seconde est anonyme et s'utilise en ligne, là où on l'écrit.

Pourquoi vouloir une fonction sans nom ?

Parce qu'on veut souvent utiliser une fonction une seule fois, sur place, sans la baptiser. C'est l'équivalent code d'un Post-it vs un livre relié : si tu n'utilises l'idée qu'une seule fois, pas la peine de la cataloguer dans un coin du fichier.

Parallèle JS : (x) => x * 2 est une arrow function. C'est exactement la même chose qu'une lambda Kotlin. Même concept, syntaxe différente.

La graphie { params -> corps } décortiquée

La graphie d'une lambda Kotlin se lit en trois morceaux :

{           x          ->          x * 2
│           │           │             │
│           │           │             └── corps : ce que la lambda exécute / renvoie
│           │           │
│           │           └── sépare paramètre(s) et corps (se lit "renvoie vers")
│           │
│           └── nom donné au paramètre que la lambda VA RECEVOIR
│
└── j'ouvre une fonction-sans-nom (pas un bloc d'instructions classique)

Donc { x -> x * 2 } se lit, mot pour mot : « une fonction qui prend x et renvoie x * 2 ».

Cas particuliers de la graphie

CasGraphieLecture
Aucun paramètre { println("Hello") } Pas de -> du tout. La lambda ne reçoit rien et fait juste son corps.
Un seul paramètre { it * 2 } Raccourci : on peut omettre le x -> et utiliser le mot-clé implicite it.
Plusieurs paramètres { a, b -> a + b } Liste de paramètres séparés par des virgules à gauche du ->.

C'est pour cette raison que dans Compose tu vois souvent { innerPadding -> ... } ou { tab -> ... } : ce sont juste des lambdas qui annoncent qu'elles vont recevoir un paramètre et lui donnent un nom local.

Pourquoi : passer du comportement en argument

Voici la vraie raison pour laquelle Kotlin a inventé ça. Avant les lambdas, on pouvait passer à une fonction des données (un nombre, une chaîne, une liste). Avec les lambdas, on peut aussi lui passer un comportement, une action, du code.

Exemple parlant : map sur une liste

val nombres = listOf(1, 2, 3)
val doubles = nombres.map { x -> x * 2 }   // → [2, 4, 6]

Que se passe-t-il ? La fonction map ne sait pas à l'avance ce qu'elle doit faire à chaque élément. C'est toi qui lui dis, en lui passant une lambda : « pour chaque élément, fais ÇA ». La même map peut servir à doubler, à mettre en majuscules, à formater des dates, à n'importe quoi — parce que le comportement est paramétrable.

Analogie : map est une machine à laver universelle. Pour qu'elle sache quoi faire à tes vêtements, tu lui glisses un programme (la lambda). Demain tu lui glisses un autre programme, elle fera autre chose. La machine ne change pas.

Lien direct avec Compose : les slots

C'est exactement le mécanisme des slots Compose (cf. article Slot Compose = lambda passé en paramètre). Un Scaffold ne sait pas à l'avance ce que tu veux dans sa barre du bas. Donc tu lui passes une lambda qui, quand elle sera exécutée, dessinera ta AppBottomBar :

Scaffold(
    bottomBar = { AppBottomBar(navController, currentRoute) }
)

bottomBar est un paramètre de type lambda. Tu lui donnes une fonction-sans-nom qui dit « quand tu auras besoin de la barre du bas, exécute ce code ». Le slot, c'est juste un paramètre lambda.

Le trailing lambda — sucre syntaxique

Kotlin a une règle de confort qu'il faut connaître par cœur :

Si le dernier argument d'une fonction est une lambda, on peut le sortir des parenthèses.

C'est purement esthétique. Ces deux écritures sont strictement équivalentes :

// Forme stricte : la lambda est dans les parenthèses
nombres.map({ x -> x * 2 })

// Forme avec trailing lambda : la lambda est sortie dehors
nombres.map { x -> x * 2 }

Et si la lambda est le seul argument, on peut même supprimer les parenthèses vides. C'est pour ça que nombres.map { it * 2 } est si compact.

En Compose, c'est la convention universelle pour le contenu :

Scaffold(
    topBar = { ... },
    bottomBar = { ... }
) { innerPadding ->
    // ← le contenu, dernier argument, sorti dehors
    NotesScreen(...)
}

Lu correctement : Scaffold(topBar = ..., bottomBar = ..., content = { innerPadding -> NotesScreen(...) }). C'est juste plus joli avec le content sorti dehors.

Application : la lambda { innerPadding -> … } du Scaffold

« Prends le code du Scaffold de mon MainScreen et explique-moi la lambda avec innerPadding. »

Voici le vrai bloc tel qu'il est dans MainActivity.kt :

Scaffold(
    modifier = Modifier.fillMaxSize(),
    bottomBar = {
        if (showBottomBar) {
            AppBottomBar(navController, currentRoute)
        }
    }
) { innerPadding ->
    Box(modifier = Modifier.padding(innerPadding)) {
        AppNavGraph(navController)
    }
}

1. Repère d'abord les deux lambdas

LambdaOùRôle
bottomBar = { … }dans les parenthèses« voici ce que tu dessineras dans le slot du bas »
{ innerPadding -> … }après la parenthèse fermante« voici ce que tu dessineras dans le slot du contenu » (= le slot principal)

La deuxième est sortie dehors → trailing lambda. Au format strict, le code donnerait :

Scaffold(
    modifier = …,
    bottomBar = { … },
    content = { innerPadding -> Box(...) { AppNavGraph(navController) } }
)

Le content = … est implicite quand on l'écrit en trailing.

2. Décortique { innerPadding -> … }

  • { → j'ouvre une lambda.
  • innerPadding → nom donné au paramètre que la lambda va recevoir.
  • -> → sépare paramètre et corps.
  • Box(modifier = Modifier.padding(innerPadding)) { ... } → le corps : ce que la lambda exécute.

Lecture mot pour mot : « une fonction qui reçoit un truc qu'on appellera innerPadding, et qui dessine un Box paddé avec, contenant AppNavGraph(navController). »

3. Mais qui « passe » le innerPadding à la lambda ?

C'est Scaffold lui-même. Point clé à intégrer : quand on écrit bottomBar = { AppBottomBar(...) }, la lambda ne reçoit rien (pas de paramètre ->). Quand on écrit { innerPadding -> ... }, on dit l'inverse : « je te promets que tu recevras un paramètre, et je l'appellerai innerPadding ».

Pourquoi cette promesse ? Parce que la signature du paramètre content du Scaffold ressemble à ça :

content: @Composable (PaddingValues) -> Unit

Traduction : « je veux une fonction @Composable qui prend un PaddingValues et ne renvoie rien ». Donc Scaffold doit te donner un PaddingValues quand il appellera ta lambda. Toi, tu prévois une variable pour le recueillir → c'est le innerPadding ->.

Le nom est libre. On pourrait écrire { pad -> Box(modifier = Modifier.padding(pad)) { … } } ou { trucMachin -> ... }. La convention est innerPadding parce que c'est explicite.

4. Qu'est-ce que c'est, ce innerPadding, concrètement ?

C'est un PaddingValues : un objet qui contient quatre marges (haut, bas, gauche, droite), calculées par le Scaffold en fonction de :

  • la hauteur de la barre du haut (s'il y en a une) ;
  • la hauteur de la barre du bas (la AppBottomBar quand showBottomBar est vrai) ;
  • les zones système (status bar, gesture bar du bas en edge-to-edge).

Scaffold dit : « j'ai réservé ces zones pour les barres et les insets. Si tu veux que ton contenu ne passe pas DESSOUS, applique ce padding à ton contenu. » Et c'est exactement ce que fait Box(modifier = Modifier.padding(innerPadding)) { AppNavGraph(navController) }.

Si on oubliait le padding(innerPadding) : le contenu (AppNavGraph) s'afficherait par-dessus la AppBottomBar. Les boutons de la nav seraient à moitié cachés. C'est la raison d'être de cette mécanique.

5. Le déroulé en runtime — pour bien sceller le concept

  1. Compose appelle Scaffold(modifier = …, bottomBar = …, content = …).
  2. Scaffold mesure la place disponible, dessine sa bottomBar, calcule le PaddingValues restant pour le contenu.
  3. Il appelle la lambda content en lui passant ce PaddingValues : c'est l'instant où innerPadding reçoit une vraie valeur.
  4. La lambda exécute Box(modifier = Modifier.padding(innerPadding)) { AppNavGraph(navController) }.
  5. → AppNavGraph est dessiné dans la zone restante.

On n'écrit jamais cet appel soi-même. Scaffold le fait en coulisses. C'est la raison d'être de la lambda : permettre à Scaffold de t'injecter une valeur (innerPadding) au moment où il en aura besoin.

En une phrase : le innerPadding -> n'est pas une variable que toi tu remplis. C'est un trou que tu prévois pour que Scaffold t'envoie quelque chose. La lambda dit : « quand tu m'appelleras, tu me donneras un PaddingValues, et je l'appellerai innerPadding ». Tu fais ensuite ce que tu veux avec — ici, tu l'appliques en padding sur un Box pour que le contenu respecte la zone de la bottomBar.

Même mécanique que nombres.map { x -> x * 2 } : map te passe chaque élément, tu le récupères dans x, tu fais ton calcul. Sauf qu'ici, le « truc passé » est un PaddingValues, et le « calcul » est de dessiner ton arborescence Compose.
